# 双向聊天机器人 PRD（产品需求文档）

> 文档版本：v0.2
> 最后更新：2026-06-16
> 负责人：项目所有者（你）
> 项目代号：twochatbot

---

## 1. 背景与问题

Telegram 自带私聊不便于"匿名 / 集中 / 可管理"地对外接收消息：
- 直接私聊会暴露个人账号，且消息散落、不易管理。
- 现有的双向机器人能转发，但**垃圾广告消息泛滥**——经常有人通过双向机器人发广告，人工屏蔽很烦。

因此需要一个**部署在 Cloudflare 上、带 AI 智能过滤**的个人双向聊天机器人：把陌生人消息安全地转发给你（管理员）、你可直接回复，同时用 AI 自动识别并拦截广告 / 垃圾 / 诈骗消息。

## 2. 目标（Goals）

- G1：实现稳定的 Telegram 双向消息中继（用户 ⇄ 管理员个人）。
- G2：接入 AI（用户自有中转站，标准 OpenAI `chat/completions`）对入站消息做智能分类与过滤，自动拦截广告 / 垃圾 / 诈骗。
- G3：全部署在 Cloudflare（Workers + KV），零服务器、低成本。
- G4：新用户首次对话需通过人机验证，降低机器人/批量广告骚扰。
- G5：管理员可控（查看被拦截内容、放行、拉黑、解封、调整过滤力度）。
- G6：AI 兼任管理员的私人助理（`/ai` 对话）与“代笔”（按管理员意向生成对陌生人的回复）；陌生人**永不直接与 AI 对话**。

### 非目标（Non-Goals，本期不做）
- 管理群 / 话题模式**保留为可选能力但本期不启用**（`RELAY_MODE=group` 预留）；默认个人私聊模式。
- 不做多客服协作。
- 不做网页后台管理面板（管理操作走 Telegram 命令 / 内联按钮）。
- 不做语音转写、图片 OCR 等重 AI 能力（后续迭代）。

## 3. 目标用户与角色

- **管理员（你，单人）**：直接在与机器人的私聊中接收转发消息、回复、管理黑名单与过滤策略。
- **普通用户（陌生人）**：通过机器人私聊向你发消息，首次需通过人机验证。陌生人只会收到：门卫拦截（或静默）、前台的一句身份问候、以及你本人或你授意 AI 代写的回复。**陌生人不能与 AI 自由对话。**

### AI 的三种角色
- **门卫**：过滤广告/垃圾/诈骗，拦截不转发。
- **前台**：对通过验证且判定正常的陌生人，自动回一句身份说明并把原消息转发给管理员。
- **代笔 / 私人助理**：仅管理员（`ADMIN_UID`）可用 `/ai`，让 AI 按意向代写对陌生人的回复，或与 AI 自由聊天处理日常事务。

## 4. 核心场景（User Stories）

- US1：用户 `/start` 机器人 → 看到欢迎语 → 触发首次人机验证。
- US2：用户通过验证 → 发送消息 → 消息被转发到你的私聊。
- US3：你在与机器人的私聊里**回复那条被转发的消息** → 机器人把回复中继给该用户。
- US4：用户发来广告 → AI 判定为垃圾 → 不打扰你，进入"已拦截"区，你可事后查看/放行。
- US5：正常陌生人发消息 → AI 前台自动回一句身份说明（“您好，这里是 XX 的助理，消息已转达，请稍候”）→ 原消息转发给你。
- US6：你 reply 转发消息直接回复对方；或发 `/ai 婉拒他` 让 AI 按你的意向+上下文代写回复并发给对方。
- US7：你空闲时发 `/ai <问题>` 与 AI 助理自由对话（写文案、查思路等），仅你可用。
- US8：你对某用户执行"拉黑" → 该用户后续消息被静默丢弃。

## 5. 功能需求（FR）

### 5.1 双向消息中继（P0）
- FR1：用户私聊消息转发到**管理员个人私聊**；支持文本、图片、文件、表情等常见类型。
- FR2：管理员**通过"回复"被转发的那条消息**来定向回复对应用户（reply-to 绑定会话）。
- FR3：每条转发消息需携带可识别发送者的信息（昵称 + 用户ID），并维护 `转发消息 ↔ 用户` 映射。
- FR4：支持管理员主动按用户ID发起会话（命令形式，可选）。

### 5.2 首次人机验证（P0）
- FR5：新用户首次 `/start` 或首次发消息时触发验证（数学题 / 简单问答，或 Cloudflare Turnstile 链接）。
- FR6：验证通过前，用户消息不转发给管理员。
- FR7：验证状态持久化（KV：`verified` 标记），通过后不再重复验证。
- FR8：验证方式与题目可通过配置项调整。

### 5.3 AI 智能过滤（P0，核心差异点）
- FR9：通过验证后的入站消息先经 AI 分类，输出 `{类别, 置信度, 理由}`，类别至少含：正常 / 广告营销 / 诈骗 / 垃圾骚扰。
- FR10：命中拦截类别时，消息不直接推送管理员，记入"已拦截"列表（含原文、判定理由）。
- FR11：管理员可查看已拦截消息，并对单条执行"放行 / 拉黑发送者"。
- FR12：AI 走用户自有中转站，标准 OpenAI `chat/completions`；模型、base_url、key 可配置。
- FR12.1：支持 **Cloudflare Workers AI（`env.AI`）作为降级备份**：当中转站额度耗尽/超时/报错时自动切换，保证过滤能力不中断。可配置启用与模型名。
- FR13：AI 失败 / 超时要有**多级降级策略**：中转站 → Workers AI → 关键词规则 → 默认放行并标注"未过滤"；任何一级失败都不得阻断正常消息。

### 5.6 AI 前台 / 代笔 / 助理（P0~P1）
- FR-A1（前台，P1）：对通过验证且 AI 判定正常的陌生人，自动回一句可配置的身份问候（`AUTO_GREETING`），并把原消息转发给管理员。问候开关可关。
- FR-A2（直接回复，P0）：管理员 reply 转发消息 → 原样中继给对应陌生人。
- FR-A3（代笔，P1）：管理员对某会话发 `/ai <意向>` → AI 结合该会话上下文与意向，生成对该陌生人的回复，经管理员发出（可配置“直接发出”或“先给管理员预览确认”）。
- FR-A4（私人助理，P1）：管理员发 `/ai <问题>`（非会话上下文）→ AI 直接回答管理员，供日常使用；支持多轮上下文。
- FR-A5（鉴权）：所有 `/ai` 能力**仅** `from.id === ADMIN_UID` 可用；陌生人发 `/ai` 视为普通消息走过滤/转发，**不触发 AI 对话**。
- FR-A6（额度复用）：前台/代笔/助理复用多级 AI 通道（中转站→Workers AI→降级）。

### 5.4 反垃圾与管理（P1）
- FR14：黑名单（拉黑 / 解封），被拉黑用户消息静默丢弃。
- FR15：关键词规则兜底（硬拦截词 + 自动回复词），与 AI 互补。
- FR16：频率限制（同一用户单位时间消息数上限），防刷。

### 5.5 管理操作（P1）
- FR17：管理命令 / 内联按钮：拉黑、解封、查看拦截、放行、查看用户信息。
- FR18：可配置项通过环境变量管理（欢迎语、过滤开关、阈值、验证方式等）。

## 6. 非功能需求（NFR）

- NFR1：部署平台 Cloudflare Workers，存储用 KV（绑定名 `TG_BOT_KV`）。
- NFR2：Webhook 入口需校验密钥，防止伪造请求。
- NFR3：密钥类配置（Bot Token、AI Key）用 Secrets，不写进代码 / 仓库。
- NFR4：单条消息端到端处理（含 AI）目标 < 3s；AI 超时阈值可配。
- NFR5：成本可控，优先 Cloudflare 免费额度；AI 调用走自有中转站额度。
- NFR6：隐私：仅存必要数据，敏感日志可关闭。

## 7. 配置项（草案）

| 变量 | 类型 | 说明 |
|------|------|------|
| `BOT_TOKEN` | Secret | Telegram 机器人 Token |
| `BOT_SECRET` | Secret | Webhook 校验密钥 |
| `ADMIN_UID` | Var | 管理员（你）的 Telegram 用户 ID |
| `RELAY_MODE` | Var | 转发模式：`private`（默认，启用）/ `group`（预留，未启用） |
| `ADMIN_GROUP_ID` | Var | 管理群 ID（仅 `group` 模式需要，以 -100 开头；本期留空） |
| `AI_BASE_URL` | Var | 中转站 base_url（标准 OpenAI 兼容） |
| `AI_API_KEY` | Secret | 中转站 key |
| `AI_MODEL` | Var | 中转站使用的模型名 |
| `AI_PROVIDER` | Var | AI 策略：`relay`（仅中转站）/ `workers_ai`（仅 CF）/ `auto`（主备自动切换，默认推荐） |
| `AI_FALLBACK_TO_CF` | Var | 中转站失败时是否回落 Workers AI（默认 `true`） |
| `CF_AI_MODEL` | Var | Workers AI 模型名（默认 `@cf/meta/llama-3.1-8b-instruct`） |
| `AI_TIMEOUT_MS` | Var | AI 超时阈值（默认 2500） |
| `FILTER_ENABLED` | Var | AI 过滤总开关 |
| `FILTER_THRESHOLD` | Var | 判定为垃圾的置信度阈值 |
| `VERIFY_MODE` | Var | 验证方式：`math` / `quiz` / `turnstile` |
| `VERIFY_QUESTION` / `VERIFY_ANSWER` | Var | 自定义验证题与答案（quiz 模式） |
| `BLOCK_KEYWORDS` | Var | 硬拦截关键词 |
| `WELCOME_MESSAGE` | Var | 欢迎语（/start） |
| `AUTO_GREETING` | Var | 前台自动身份问候语；留空则不自动回复 |
| `AI_REPLY_PREVIEW` | Var | 代笔模式：`send`（直接发出）/ `preview`（先给管理员确认，默认） |
| `AI_CONTEXT_ROUNDS` | Var | AI 多轮上下文保留轮数（默认 6） |

## 8. 验收标准（MVP）

- AC1：陌生人 → 机器人 → 管理员私聊链路通；管理员"回复转发消息"能回到对应用户。
- AC2：新用户首次必须通过验证才能把消息送达管理员。
- AC3：构造一条典型广告，AI 能拦截且不打扰管理员；管理员能在拦截列表看到并可放行。
- AC4：正常消息不被误拦（提供放行兜底）。
- AC5：AI 故障时正常消息不丢、不阻塞。
- AC6：一键拉黑后该用户消息不再转发。
- AC7：正常陌生人能收到前台身份问候，且原消息转发到管理员。
- AC8：管理员 `/ai <意向>` 能基于会话上下文代写回复；陌生人发 `/ai` 不触发 AI 对话。
- AC9：管理员 `/ai <问题>` 能与助理多轮对话。

## 9. 里程碑

- M1 双向中继跑通（个人私聊，无 AI，无验证）。
- M2 首次人机验证 + AI 过滤 + 拦截列表 + 放行/拉黑。
- M3 反垃圾增强（频率限制、关键词）+ 配置化打磨。

## 10. 已确认决策

- D1：默认**个人单管理员私聊转发**（`RELAY_MODE=private`），回复方式 = 对被转发消息使用 Telegram「回复」。管理群/话题模式（`RELAY_MODE=group`）**保留为可选能力，本期不启用**，未来配置 `ADMIN_GROUP_ID` 即可开启，无需改代码。
- D2：存储采用 **Cloudflare KV**（后续如需全文检索/统计再评估 D1）。
- D3：AI 接口为**标准 OpenAI `chat/completions`**。
- D4：**启用新用户首次人机验证**。
- D5：AI 采用**多级策略**：主用自有中转站，**Cloudflare Workers AI 作为降级备份**（中转站额度耗尽/故障时自动接管），最终兜底关键词规则。
- D6：AI 定位为**门卫(过滤)+前台(身份问候)+代笔/私人助理**；**陌生人不与 AI 自由对话**，主动权始终在管理员。取消早期“陌生人自助 AI 客服(方案 B)”设想。
